Hashing Collision - Separate Chaining versus Linear Probing

Hashing functions are not perfect. Sometimes you might run into hashing collision where different objects are hashed into the same location - hashing collision. There are ususally two different ways to solve this problem - separate chainig or probing. Separate chaining is just linked all objects that are hashed to the same location as a linked list whereas probing place the object at the next avaible spot. It is very counter intuitive when I first learnt that probing, even linear probing, can be more time efficient than separate chaining. As a stubborn student, I decided to analyse runtime of the two methods.



The plot above clearly indicates that as sample size increases, it requires more reads to perform search operation with separate chaining whereas number of reads don't seem to change much with linear probing. The trick to linear probing is that each time we increase the capacity of the hashed table, we redistribute the previous elements in a more scattered fashion so that there will be more spaces between elements to store any hashing-collided objects.

Note that linear probing is not perfect because more memory is needed than with separate chaining.

Welcome to the world of algorithms; there is no magical unicorn algorithm - everything is a trade-off.
